# Captured session 2026-05-05

最初のフィールド試験で採取した AMB P3 デコーダーのバイト列。
`gateway/testdata/captured/session-2026-05-05.bin` として配置されている匿名化済みフィクスチャの素性を記録する。

## セッション概要

| 項目 | 値 |
|------|------|
| 日時 | 2026-05-05 11:35 JST 〜(数十秒〜分のオーダー、終了時刻は不明) |
| 場所 | フィールド(屋外コース) |
| 上流 AMB | 192.168.1.21:5403 |
| 採取側 OS | Windows |
| ゲートウェイ EXE | `field-2026-05-05-c211faddeb6d`(`docs/incidents/2026-05-05-recorder-csv-flush.md` 参照、当時はバグあり) |

採取者は本人で、**走行は採取者以外の第三者**(本テストではテスト中に走行できないため、これは想定どおり)。

## 採取結果(匿名化前→後)

| 項目 | 採取生データ(`dist/...`、コミット禁止) | コミット済み(`gateway/testdata/captured/session-2026-05-05.bin`) |
|------|--------|--------|
| サイズ | 2102 byte | 2097 byte |
| frame 数 | 58 | 58 |
| TOR 分布 | 同左 | TOR 0x0001 (PASSING) ×15、TOR 0x0002 (STATUS) ×42、TOR 0x001C(undocumented)×1 |
| TRANSPONDER 値 | 第三者の実 ID 2 種 | **synthetic `0x00000001`, `0x00000002`(観測順)** |
| PASSING_NUMBER | `0xE3`〜`0xF1` 連番、欠番なし | 同左(変更なし) |
| RTC_TIME | 単調増加 | 同左(変更なし) |
| STRENGTH / HITS / FLAGS | 同上 | 同左(変更なし) |
| ヘッダの CRC | original | 同左(再計算しない、`docs/protocol-p3.md` §6 方針) |
| ヘッダの FrameLength | original | 同左(再計算しない。`0x52998D` のように生 transponder に `0x8D` を含む 5 frame は wire 長が 1 byte ずつ短くなるが、FrameLength フィールドは original の値を保持) |

サイズ差 5 byte は、escape 対象バイトを含む transponder(`0x0052998D` の 0x8D)が再エンコード時に escape されないためで、frame 5 件 × -1 byte = -5 byte。

## 匿名化の実体

`gateway/cmd/anonymize`(本リポジトリ)が以下を実行:

1. P3 frame split(0x8E … 0x8F 境界)
2. unescape
3. TOR == 0x0001 (PASSING) のフレームのみ、TLV 走査で id=0x03 (TRANSPONDER, 4 byte LE) を検出 → 観測順の synthetic ID(1, 2, …)に置換
4. re-escape

実行コマンド:

```pwsh
go run .\gateway\cmd\anonymize `
    -in  "..\dist\AMB_RC_Lap_Timer\records\session-2026-05-05.bin" `
    -out "gateway\testdata\captured\session-2026-05-05.bin"
```

匿名化前→後の TRANSPONDER マッピング表は **本ファイルにも、リポジトリ内のどこにも書かない**。クロスセッション相関が必要な場合のみ、操作員が手元のローカルメモで保管する。

## 同セッションの timing.csv について

本セッションでは **timing.csv は破損しており復元不可**。経緯と修正は `docs/incidents/2026-05-05-recorder-csv-flush.md` 参照。
このため `.timing.csv` はコミットしない。`gateway/testdata/captured/session-2026-05-05.bin` はタイミング情報を持たないが、byte 構造テスト・フィルタロジックテスト・RTC_TIME 単位推定 にはそのまま利用できる。

## 既知の解析メモ

- 1 frame 目の TOR は `0x001C`(undocumented)。body は `0x01` (8 byte) と `0x81` (DECODER_ID, 4 byte) の TLV のみで、AMB 接続時の挨拶系レコードと推定。`docs/protocol-p3.md` §9 オープン項目に追加候補。
- 2 種の transponder の出現順は synthetic 1 → 2 → 1 → 2 …。連番 `PASSING_NUMBER` から、両ポンダーがほぼ同時にループを通過していたことが分かる。
- RTC_TIME は単調増加で、ラップタイム計算用途には十分(単位は §9 #3 オープン項目)。
