# Gateway Technical Decision (2026-05-04)

## 背景
- 本プロジェクトは、スマートフォン/タブレットのWebブラウザで利用する個人練習向けラップタイマーを目指す。
- AMB P3 デコーダーはLAN上にあり、ブラウザ側では低遅延な表示/音声読み上げを実現したい。
- 利用現場にはWindows 8/11のPCが存在するが、管理人の運用負荷は最小化する必要がある。
- Windows 11マシンは会計用途もあるため、常駐アプリ追加のリスクは極力避ける。

## 議論の結論

### 1. 配置方針
- **ゲートウェイは「LAN内のWindows PC」で動作可能とする。**
- ただし運用上は、会計用途PCと分離できるなら分離を推奨。

### 2. 技術スタック（推奨）
- **第一候補: Go製の単体EXEゲートウェイ**
  - 上流: AMB P3へTCPクライアント接続
  - 下流: WebSocketでブラウザへ配信
  - 配布: `gateway.exe`（設定は`config.json`）
  - ログ: ローカルファイル + ローテーション
  - 監視/復旧: 自動再接続（バックオフ付き）

### 3. なぜGoを推奨するか
- 単体EXE配布に向き、現場で依存解決がほぼ不要。
- TCP/WebSocket中継の実装がシンプルで保守しやすい。
- 再接続・監視・ログ等の運用要件を実装しやすい。

### 4. 非推奨/優先度低
- Python: 実行環境依存・配布運用コストの観点で不利。
- .NET新しめランタイム依存: 古いWindows互換の確実性が課題。
- Rust: 技術的には有力だが、初期開発/保守難度が相対的に高い。

## 実機AMBなし（手元はWindows 11のみ）での開発方針
- 受信ソースを抽象化し、以下を差し替え可能にする。
  1. 実機TCP入力
  2. Mock TCPサーバ入力
  3. ログリプレイ入力
- これにより、実機がなくても実装/検証を継続できる。

## 初期実装の責務分離（別エージェント向け）
- **バックエンド**
  - AMB TCP受信
  - フレーム/レコードのパース
  - WebSocket配信
  - 再接続、ログ、ヘルス状態公開
- **フロントエンド**
  - WebSocket受信
  - ポンダー別ラップ表示
  - 音声読み上げ
  - ローカル設定保存（対象ポンダー、音量、読み上げ設定）

## 直近の実装優先順位
1. Mock TCP + WebSocket配信の最小疎通
2. フロントのリアルタイム表示
3. 音声読み上げ
4. 切断復旧とログ整備
5. 実機接続テスト

## 備考
- 本決定は「現時点の運用制約（Windows中心・低運用負荷）」を優先したもの。
- 将来的には専用ミニPCやLinuxゲートウェイへの移行余地を残す。
