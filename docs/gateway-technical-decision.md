# Gateway Technical Decision (2026-05-04)

## 背景
- 本プロジェクトは、スマートフォン/タブレットのWebブラウザで利用する個人練習向けラップタイマーを目指す。
- AMB P3 デコーダーはLAN上にあり、ブラウザ側では低遅延な表示/音声読み上げを実現したい。
- ブラウザは生TCPを叩けないため、LAN内に薄い中継(ゲートウェイ)を置く必要がある。
- 利用現場にはWindows 8.1/11のPCが存在するが、管理人の運用負荷は最小化する必要がある。
- Windows 11マシンは会計用途もあるため、常駐アプリ追加のリスクは極力避ける。

## 議論の結論

### 1. 配置方針
- ゲートウェイは「LAN内のWindows PC」で動作させる。
- 運用上は、会計用途PCと分離できるなら分離を推奨。
- 将来は中古ミニPC(Linux)へ移行できるよう、OS依存を最小化する。

### 2. 配信トポロジ
- **ゲートウェイEXEが UI(SPA)も同梱配信する。** 外部Webホスティングは使わない。
  - SPAは `go:embed` でEXEに同梱し、`http://<gw-ip>:<port>/` で配信。
  - 同一プロセス内のWebSocketエンドポイント `ws://<gw-ip>:<port>/ws` でラップデータを配信。
  - 配布物は `gateway.exe` + `config.json` の2点のみ。
- これによりブラウザ側の Mixed Content 問題と外部依存を同時に回避する。
- スマホからは固定IP(推奨)または `http://<gw-host>.local/` (mDNS可なら)で到達。

### 3. 技術スタック(推奨)
- **Go 製の単体EXEゲートウェイ**
  - **Goバージョン: 1.20.x で固定**(Windows 7/8.1サポートを保持する最終メジャー)。
  - 上流: AMB P3 デコーダーへのTCPクライアント接続。
  - 下流: SPA静的配信 + WebSocket fan-out(複数クライアント同時接続を許容)。
  - 設定: 起動時 `config.json` 読み込み + 起動後は `/admin` の設定WebUIから編集。
  - ログ: ローカルファイル + ローテーション、`/logs` で直近を閲覧可能にする。
  - 監視/復旧: 自動再接続(初期1s → 最大30s、指数バックオフ + ±20% jitter)、`/healthz` でヘルス公開。

### 4. パース責務の置き場所
- **AMB P3 のフレーム/レコードのパースは「フロントエンド(TypeScript)」で実施する。**
- ゲートウェイは TCP ↔ WebSocket の **薄い byte pipe** に徹する。
  - メリット: プロトコル微差や新フィールドが出てもEXE再配布が不要、Web更新だけで反映可能。
  - メリット: パーサのテスト/デバッグが DevTools で完結。
  - 帯域は LAN 内通信のため、JSON化していない素のバイト列でもボトルネックにならないと判断。

### 5. なぜGoを推奨するか
- 単体EXE配布に向き、現場で依存解決がほぼ不要。
- TCP/WebSocket中継、静的配信(`embed`)、再接続、ログまで標準ライブラリで完結。
- Windows 8.1 対応の最後のメジャー(1.20.x)が安定して残っている。

### 6. 非推奨/優先度低
- Python: 実行環境依存・配布運用コストの観点で不利。
- .NET 新しめランタイム依存: 古いWindows互換の確実性が課題。
- Rust: 技術的には有力だが、初期開発/保守難度が相対的に高い。

### 7. Windows 互換性とセキュリティ方針
- **動作対象: Windows 8.1 (x64) 以上**。Windows 8 RTM は対象外。
- Windows 10/11 を推奨、Windows 8.1 はベストエフォート扱い。
- Go 1.20系は2024年に上流のセキュリティ修正が終了している点を明記する。
- LAN内専用とし、**外部公開・インターネット越し利用は禁止**。TLS/認証は実装しない(LAN信頼前提)。
- Windows Defender Firewall の受信許可は初回起動ガイドで案内する。

## 実機AMBなし(手元はWindows 11のみ)での開発方針
- 受信ソースを抽象化し、以下を差し替え可能にする。
  1. 実機TCP入力
  2. Mock TCPサーバ入力(`gateway.exe --mock`)
  3. ログリプレイ入力(`gateway.exe --replay <file>`)
- 実機ログのキャプチャは `gateway.exe --record <file>` で生バイト列を保存。
- リプレイ時は等速/早送り/即時を切り替え可能とする。
- これにより、実機がなくても実装/検証を継続できる。

## 初期実装の責務分離(別エージェント向け)
- **ゲートウェイ(Go)**
  - AMB デコーダーへの TCP 接続と再接続制御
  - WebSocket fan-out(受信バイト列を全クライアントへ透過配信)
  - SPA の静的配信(`go:embed`)
  - 設定 WebUI(`/admin`)、ヘルス公開(`/healthz`)、ログ閲覧(`/logs`)
  - Mock / Replay / Record モード
- **フロントエンド(TypeScript)**
  - WebSocket 受信と P3 フレーム/レコードのパース
  - ポンダー別ラップ表示
  - 音声読み上げ(Web Speech API)
  - ローカル設定保存(対象ポンダー、音量、読み上げ設定など)

## 運用要件
- **自動起動**: タスクスケジューラのログオン時実行 or スタートアップショートカットで対応(サービス化はしない)。
- **ファイアウォール**: 初回起動時の Windows Defender Firewall 許可手順を README に記載。
- **到達性**: 固定IP運用を基本とする。mDNS(`*.local`)はクライアントOSによる差があるため補助扱い。
- **設定変更**: `config.json` 直編集ではなく `/admin` の WebUI を一次手段とする。
- **複数クライアント**: スマホ+タブレット同時接続を想定。WS は fan-out 設計で1接続限定にしない。

## 直近の実装優先順位
1. ゲートウェイ最小骨格(SPA同梱配信 + WS fan-out + Mock TCP)で疎通
2. フロントの P3 パーサとリアルタイム表示
3. 音声読み上げ
4. 切断復旧、ヘルス公開、ログ整備、設定WebUI
5. Record/Replay 整備
6. 実機接続テスト

## 備考
- 本決定は「現時点の運用制約(Windows中心・低運用負荷)」を優先したもの。
- 将来的には専用ミニPCやLinuxゲートウェイへの移行余地を残す(OS依存を最小化する設計を維持)。
